# IP验证登录系统 - 项目详细说明

## 1. 项目概述
本项目是一个面向小规模私有使用场景的认证系统，服务个人 Web 应用及少量可信用户。系统通过账号密码与 TOTP 进行强认证，并结合 IP 与 Cookie 降低重复登录频率，最终对接 Nginx `auth_request` 实现统一访问控制。

## 2. 技术选型
- 编程语言：Python 3.11+
- 数据库：SQLite（内置 `sqlite3`）
- 服务形态：HTTP 鉴权服务（可被 Nginx `auth_request` 调用）
- 部署方式：单进程/单机优先，适合小规模使用
- 接口命名约定：认证与登录相关接口统一使用 `/auth/*` 前缀，避免与业务路由冲突

## 3. 建设背景与问题
中国大陆家宽公网 IP 常为动态分配，固定白名单不可行。若仅依赖账号密码 + TOTP，会造成高频重复登录。项目通过“用户/IP/Cookie/地点”动态信任模型平衡安全和体验。

## 4. 项目目标
- 实现账号密码 + TOTP 双因子认证。
- 满足条件时允许免登录放行，减少交互成本。
- 支持多用户、多站点场景。
- 支持地点管理并引入“公共场所”加固策略。
- 提供 Nginx `auth_request` 所需鉴权接口。

## 5. 角色与使用对象
- 普通用户：登录系统、选择地点、访问业务站点。
- 管理员：管理地点信息、查看 IP 与地点绑定关系。
- Nginx：调用鉴权接口，根据返回状态决定放行或拦截。

## 6. 业务规则（核心）
### 6.1 标准规则
1. 相同 IP 且 Cookie 正确且有效：直接放行。
2. 相同 IP 且 Cookie 过期：放行。
3. 相同 IP 且无 Cookie 或 Cookie 错误：密码或 TOTP 二选一。
4. 不同 IP 且 Cookie 相同：密码或 TOTP 二选一，并强制重新选择该 IP 所在地点。
5. 不同 IP 且 Cookie 过期/无 Cookie/Cookie 错误：密码 + TOTP 同时验证。

### 6.2 公共场所加固规则
地点若标记为“公共场所”，则在“相同 IP + Cookie 过期”场景下，从直接放行升级为“密码或 TOTP 二选一”。

## 7. 功能需求
### 7.1 登录与验证
- 用户名/密码登录。
- TOTP 动态口令验证。
- 首次新 IP 或不可信状态触发挑战。
- 挑战类型由后端策略自动决定，前端不可手动选择。
- 登录成功后再进行地点选择（已有地点或新建地点）。

### 7.2 地点管理
- 地点字段：名称、是否公共场所。
- 支持地点增删改查。
- 支持当前 IP 与地点绑定。

### 7.3 管理面板数据
- 地点列表。
- IP-地点绑定关系。
- 登录/鉴权审计记录（用户、IP、地点、时间、验证方式、结果）。

### 7.4 Nginx 对接
- 提供 `GET /auth/check`。
- 通过 2xx（放行）与 401/403（拦截）表达鉴权结果。

## 8. 非功能需求
- 安全性：密码哈希存储、TOTP 校验、Cookie 防篡改。
- 可用性：鉴权接口低延迟。
- 可观测性：关键链路日志可追踪。
- 可维护性：策略逻辑可配置和可测试。

## 9. 数据记录要求
关键行为至少记录：用户 ID、源 IP、地点 ID、时间、验证类型、验证结果、站点标识。

## 10. 验收标准
- 完整实现 5 条标准规则 + 1 条公共场所加固规则。
- 支持多用户、多站点接入。
- 可管理地点并查看 IP-地点绑定。
- 可与 Nginx `auth_request` 联调通过。
- 日志可追踪一次完整认证决策。

## 11. 当前实施状态
已进入实施阶段，当前代码基线为：
- Python + SQLite 基础工程
- 数据库初始化脚本
- 认证规则引擎
- 核心接口（`/auth/check`、`/auth/login`、`/auth/locations`、`/auth/bindings`）
